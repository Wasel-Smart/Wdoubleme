A Figma iframe is showing an error because the message port is destroyed during communication. Analyze the current iframe setup, identify why the message port might be closing unexpectedly, and provide a robust solution to handle message port lifecycle safely. Include steps to:The error "message port is destroyed" (or very similar phrasing like "The message port is destroyed during communication", "port is closed/destroyed", or related MessagePort exceptions) in a Figma iframe context almost always relates to how Figma plugins handle communication between:

The plugin code (code.ts / main script — runs in Figma's sandboxed VM)
The UI (either the built-in HTML <iframe> created via figma.showUI(…), or — more relevant to your case — an externally hosted custom iframe you load via figma.showUI('<iframe src="https://your-domain.com/…" …>') or similar)

Figma internally uses a pair of entangled MessagePort objects (one in the plugin VM, one in the UI iframe) to power figma.ui.postMessage(…) ↔ window.onmessage / parent.postMessage(…).
When that internal MessagePort gets closed or garbage-collected unexpectedly, any pending or new messages fail with exactly this family of errors.
Most Common Root Causes in 2025–2026 Figma Ecosystem

The plugin VM / UI iframe is being destroyed / unmounted while a message is still in flight
→ Most frequent when the user closes the plugin window (or Figma forces it closed) right after you sent a message but before the reply arrives.
React strict mode + double mount/unmount in development
→ In React dev mode (<React.StrictMode>), components mount → unmount → mount again → your useEffect that calls parent.postMessage or sets up window.onmessage may run twice, or clean up too aggressively, breaking the port timing.
External iframe navigation / reload / React remount
→ If your external React app does a client-side navigation (e.g. react-router → new route), or a state change causes full re-render/remount of the root component → the old iframe content (and its MessagePort) is destroyed → any pending response from Figma arrives too late → destroyed port error.
Long-running async operations without checking if UI is still alive
→ You send a message → wait 5–30 seconds for heavy computation in code.ts → meanwhile user closes plugin → port dies → reply fails.
Multiple / duplicate message listeners or ports
→ Accidental duplicate window.onmessage = … assignments or multiple figma.ui.onmessage handlers → Figma sometimes closes older ports.

Robust Solution: Safe Message Port Lifecycle Handling
Follow these layered defenses:
1. Use a request–response pattern with unique IDs + timeout + alive flag (strongly recommended)
In your UI code (React / external iframe):
tsx// ui.tsx / App.tsx
import { useEffect, useRef, useState } from 'react';

type MessageId = string;

interface PendingRequest {
  resolve: (value: any) => void;
  reject: (err: Error) => void;
  timeoutId: NodeJS.Timeout;
}

export default function App() {
  const pending = useRef<Map<MessageId, PendingRequest>>(new Map());
  const isMounted = useRef(true);

  useEffect(() => {
    // Cleanup flag
    return () => {
      isMounted.current = false;
      // Optional: reject all pending on unmount
      pending.current.forEach(p => p.reject(new Error("UI unmounted")));
      pending.current.clear();
    };
  }, []);

  const sendToFigma = (payload: any): Promise<any> => {
    if (!isMounted.current) {
      return Promise.reject(new Error("UI is already unmounted"));
    }

    const id = crypto.randomUUID(); // or Date.now() + Math.random()
    const promise = new Promise((resolve, reject) => {
      const timeoutId = setTimeout(() => {
        pending.current.delete(id);
        reject(new Error("Figma response timeout (port likely destroyed)"));
      }, 15000); // adjust per your needs

      pending.current.set(id, { resolve, reject, timeoutId });
    });

    // Very important: use '*' in production external iframes
    parent.postMessage({ pluginMessage: { ...payload, __id: id } }, "*");

    return promise;
  };

  useEffect(() => {
    const handler = (event: MessageEvent) => {
      // Basic security: in production you should check event.origin if possible
      const msg = event.data?.pluginMessage;
      if (!msg || !msg.__id) return;

      const req = pending.current.get(msg.__id);
      if (!req) return;

      clearTimeout(req.timeoutId);
      pending.current.delete(msg.__id);

      if (msg.error) {
        req.reject(new Error(msg.error));
      } else {
        req.resolve(msg.data ?? msg);
      }
    };

    window.addEventListener("message", handler);
    return () => window.removeEventListener("message", handler);
  }, []);

  // Example usage
  const fetchSomething = async () => {
    try {
      const result = await sendToFigma({ type: "get-selection-data" });
      console.log("Got data:", result);
    } catch (err) {
      console.error("Safe error:", err.message);
      // → "UI unmounted" or "timeout" instead of raw port destroyed
    }
  };

  // …
}

2. Additional Defenses

Avoid re-mounting the root component
Use React Router with basename or hash router instead of full page reloads / route changes that destroy the iframe content.
Debounce / throttle heavy postMessage calls
Prevent flooding when user types/resizes rapidly.
